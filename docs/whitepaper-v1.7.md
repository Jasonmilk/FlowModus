## 前言：为什么需要 FlowModus

2026年，LLM API市场已高度碎片化。超过100家供应商提供闭源与开源模型，每家都有独特的计费模式、API格式和缓存机制。企业CFO开始强制要求采购带有明确成本护栏的AI基础设施。然而，由于缺乏跨供应商的标准化计费声明，企业FinOps团队无法按模型、按任务类型、按部门精确归因AI成本。行业正在陷入毫无意义的内卷——各家LLM供应商在计费算法、KV Cache参数、API格式上各自为战，开发者在碎片化的生态中疲于适配，企业用户被锁定在单一供应商的封闭花园中。

**市场需要一个真正中立的第三方。**

FlowModus不是调度器，而是**调度规则的定义者**。它不控制流量，只制定标准。它不推荐任何供应商，只提供确定性的调度规则引擎。它不属于任何LLM供应商，它属于社区。

FlowModus通过开放、透明的供应商JSON声明，让企业首次获得跨供应商的标准化计费数据。其使命是终结这场内卷，让LLM供应商专注于提升模型质量和服务水平，让开发者拥有真正的自由选择权，让企业用户摆脱供应商锁定的恐惧。


## 第一章：哲学根基——中立、确定性、去中心化

### 1.1 中立是唯一的护城河

FlowModus是唯一的、真正独立的第三方。它不属于任何LLM供应商，不属于任何云服务提供商，不属于任何商业利益集团。它属于社区。这份中立性是FlowModus唯一的护城河。代码可以被复制，平台可以被取代，但社区共识不可复制。

### 1.2 确定性的两层定义

FlowModus的“确定性”分为两个严格隔离的层次：

**第一层：静态规则确定性。** 计费模型、KV Cache能力、参数范围、语义路由标签——这些是供应商声明的、变动频率极低的数据。调度引擎基于这些数据做出的决策是绝对的、可复现的。这是FlowModus确定性承诺的核心。

**第二层：动态状态自适应。** 延迟、可用性、健康状态——这些是实时波动的数据。FlowModus将这些数据用于辅助决策（如熔断降级、健康度排序），但这些数据不参与核心调度规则的计算。动态状态的引入有明确的熔断阈值——这是确定性的If-Else逻辑，而非概率模型。

### 1.3 去中心化——不拥有服务器是最大的力量

FlowModus不需要中心化服务器。每个用户运行自己的Sidecar实例，所有调度决策在本地完成。供应商JSON文件通过IPFS/IPNS协议进行去中心化分发，通过Ed25519数字签名确保文件的完整性和来源真实性。API密钥永远只存储在用户本地，绝不通过网络传输。

这种架构确保：即使官方仓库被攻击或下线，任何用户都可以基于已有代码继续运行自己的FlowModus实例；供应商退出不影响生态正常运作。真正的韧性来自于分布式的信任，而非任何单一实体的承诺。

### 2.3 多语言与Unicode规范化

FlowModus协议原生支持全球任何语言的Unicode字符。为确保密码学签名在所有平台上的“字节级绝对一致性”，所有供应商JSON声明在签名前必须经过以下规范化处理：

1.  **UTF-8编码**：所有JSON文件必须强制使用UTF-8编码，无BOM。
2.  **JSON规范化**：序列化的JSON对象按键名进行词法排序，使用标准分隔符，移除所有不必要的空格和换行。
3.  **Unicode规范化**：对整个JSON字符串统一采用**NFC (Normalization Form C)** 规范化，确保视觉相同但编码不同的字符被转换为完全一致的字节流。

**语言策略**：
-   **协议字段（Key）**：所有协议定义的字段名必须使用英文。这些是机器优先的协议契约，不应本地化。
-   **展示内容（Value）**：任何面向人类的展示内容（如`display_name`）均允许使用任何Unicode字符，包括中文、日文、韩文等。经过NFC规范化后，它们可以安全地参与签名和验证。

### 2.4 语义路由标签与Agent角色

语义路由标签和Agent角色是确定性路由的辅助信号。当用户任务被分类为数学密集型时，调度引擎优先选择标记了`math-heavy`的模型。当Agent需要代码生成能力时，调度引擎优先选择标记了`code-generation`角色的模型。这些标签由供应商自行声明，其与实测行为的偏差通过“声明偏移量”机制公开呈现。

### 2.5 认证与非认证

认证供应商的JSON文件由协议根密钥签名分发，通过所有权验证确保信息真实性。非认证供应商同样可以正常使用——用户或社区成员自行编写JSON文件，FlowModus正常加载和调度。**所有调度权重默认值均为建议，用户始终拥有最终的覆盖权和决定权。**


## 第三章：协议根密钥与供应链安全

### 3.1 信任锚点：协议根密钥

FlowModus的去中心化信任模型建立在**Ed25519数字签名**之上。协议启动时生成一对公私钥：

-   **协议根私钥**：以多签分片形式持有，由社区多签委员会共管。任何操作需满足 N-of-M 签名阈值。
-   **协议根公钥**：硬编码在所有Sidecar客户端代码中。这是整个FlowModus网络唯一的密码学信任锚点。

**多签管理原则**：
-   协议根私钥在任何时刻都不以完整形式存在于任何单一设备上。
-   密钥分片初始分配至 5-7 名多签委员会委员，需至少 3 枚分片签名方可通过更新。
-   多签委员会的日常职责限于：审批供应商身份公钥的所有权转移、签署自动生成的注册表更新 diff。
-   协议不赋予多签委员会内容审查权或供应商优选权。

### 3.2 签名与验证流程

1.  **签名**：多签委员会对规范化的供应商JSON集合进行 Ed25519 聚合签名。签名随JSON文件一同发布。
2.  **分发**：签名后的JSON文件被上传到IPFS，获得一个CID。多签委员会使用协议根私钥分片共同签署 IPNS 链接，动态指向最新CID。
3.  **验证**：Sidecar启动时，从IPNS链接拉取最新的JSON文件和签名，使用硬编码的协议根公钥验证签名的有效性。如果文件在传输过程中被篡改，签名验证将立即失败，Sidecar拒绝加载。

### 3.3 防注入深度防御

FlowModus采取多层防御策略，确保供应商JSON声明中的任何字符串都不会成为攻击向量：

-   **沙箱解析层**：所有供应商声明先经过规范化器序列化成确定性字节流。多语言字符串在NFC规范化下被消解了任何编码差异。
-   **结构校验层**：在签名之前，用JSON Schema做白名单校验，严格限制字段类型和值范围。
-   **加密签名层**：对规范化后的字节流计算哈希并验签。任何试图通过编码差异进行的“同形异义”攻击都在此层被消解。
-   **内容安全策略层**：对用于UI展示的字段，在渲染时进行HTML实体转义或ANSI剥离，阻断XSS或终端注入攻击。
-   **审计日志与恢复**：发生降级时记录完整原始输入，便于安全审计。

### 3.4 供应商自主接管

当供应商希望自行维护其JSON声明时，可以进行平滑过渡：

1.  供应商生成自己的Ed25519密钥对，并向多签委员会提交公钥及所有权证明。
2.  多签委员会验证所有权后，在IPNS注册表中将该供应商条目的信任锚点字段替换为供应商的公钥。
3.  此后，Sidecar在验证该供应商的声明时，将使用供应商的公钥而非协议根公钥。
4.  供应商从此可以自行签名并发布更新，无需经过多签委员会。

升级的平滑性：Sidecar的验证逻辑完全不变——它只是读取信任锚点字段来决定使用哪个公钥。


## 第四章：去中心化分发基建

### 4.1 分发架构

FlowModus 的供应商声明文件通过一个三层分发架构进行传播，确保全球可访问、抗审查、且内容完整性得到密码学保证：

```
供应商 JSON 声明文件
        │
        ▼
Canonicalizer → Ed25519 聚合签名 → 签名后的 JSON 包
        │
        ▼
IPFS 上传 → 获得永久 CID
        │
        └─── IPNS 链接（多签委员会维护）→ 固定不变的家门牌
                 │
                 └─── DNSLink（registry.flowmodus.dev）
                          │
                          ▼
                  Cloudflare IPFS Gateway（全球 CDN 加速）
                          │
                          ▼
                  Sidecar 节点（拉取 + Ed25519 验签）
```

**关键设计决策：**

| 组件 | 作用 | 免信任特性 |
|:---|:---|:---|
| **IPFS** | 内容寻址的 P2P 存储 | 任何人可从任意节点获取文件，无需信任服务器 |
| **IPNS** | 多签委员会维护的不可变指针 | 只有满足 N-of-M 签名阈值才能更新指针 |
| **DNSLink** | 将 IPNS 映射到人类可读域名 | Cloudflare 提供 CDN 加速，但不拥有内容主权 |
| **Ed25519 聚合签名** | 对规范化 JSON 进行多签 | Sidecar 用硬编码公钥本地验签，不依赖任何外部权威 |

### 4.2 协议根密钥的生命周期管理

-   **生成环境**：在从未连接互联网的离线设备上生成 Ed25519 密钥对。
-   **分片存储**：私钥分片分发至多签委员会委员各自管理的安全介质中，任何单一委员无法重建完整私钥。
-   **签名操作**：委员会成员在各自离线环境完成分片签名，聚合后形成完整签名。
-   **失效预案**：若协议根私钥分片达到不可恢复阈值（例如超过 M-N 枚分片同时灭失），需重新生成密钥对，并将新的公钥硬编码到 Sidecar 的后续版本中。旧版本 Sidecar 将无法验证新签名，需要用户升级。**这一预案是协议最后的灾难恢复路径，不是日常操作。**

### 4.4 不可逆性保证

FlowModus 的分发架构提供以下不可逆性：

-   **内容不可篡改**：IPFS 的内容寻址机制确保任何修改都会产生新的 CID，旧版本永远不会被覆盖。

## 第五章：Sidecar 网关架构

### 5.1 核心功能

FlowModus Sidecar 是一个轻量级本地代理，承担以下职责：

1.  **配置拉取**：启动时从 IPNS/DNSLink 拉取最新的供应商 JSON 集合，使用硬编码的协议根公钥验证签名。
2.  **路由决策**：根据用户请求的意图（语义标签、Agent角色、Token预算等），结合供应商声明的能力和计费模型，做出确定性的路由决策。
3.  **KV Cache 优化**：根据供应商的 `kv_cache` 声明，决定是否为当前请求启用 KV Cache，并选择合适的缓存策略。
4.  **匿名数据贡献**（可选）：当用户主动开启数据共享功能时，Sidecar 仅上报三条聚合数据——可用性、延迟、声明偏移量。不上报任何原始 Prompt 或用户标识信息。

### 5.2 API 代理模式

Sidecar 以 OpenAI-compatible API 代理的方式运行：

```
用户 Agent
        │
        ▼
  POST /v1/chat/completions
        │
        ▼
  FlowModus Sidecar (本地 :8080)
        │
        ├── 读取供应商 JSON 声明
        ├── 确定路由决策
        ├── 注入 KV Cache 参数
        │
        ▼
  目标 LLM 供应商 API
```

**兼容性**：任何使用 OpenAI-compatible API 的工具只需将 `base_url` 改为 `http://localhost:8080`，即可零代码改动接入 FlowModus。

### 5.3 五层确定性路由管线

FlowModus 的 Sidecar 引擎内部，采用一个漏斗式的编译与计算管线，将原始的供应商声明数据逐步编译为确定性的路由决策。这五层架构是 FlowModus 工程确定性的核心体现。

#### 🛡️ Layer 1: 协议与度量规范层

-   **痛点**：不同供应商的 Token 定义和计费方式（按Token、按时间、按图像帧）无法直接比较。
-   **确定性工程方案**：引入 **STE (Standard Token Equivalent，标准 Token 当量)**。协议以主流开源分词器为基准，定义 1 STE。供应商在 JSON 声明中提供其分词器相对于基准的压缩率参考系数。STE 用于跨供应商的报价比较辅助标尺，不进入审计逻辑。

#### 🏭 Layer 2: 供应商原始模型层

-   **痛点**：各家定价五花八门。
-   **确定性工程方案**：这是经过 Ed25519 签名验证的去中心化 JSON 注册表。它绝对忠实、客观、不可篡改地记录供应商发布的原始价格、能力参数和合规声明，不包含任何主观评价，构成整个系统的信任底座。

#### 🧪 Layer 2.5: 确定性基准测试层

-   **痛点**：供应商的 JSON 声明可能与实际 API 行为存在偏差。
-   **确定性工程方案**：引入 **1-Token 基准探测 (Canary Probe)**。Sidecar 在低负载时段，定期发送携带唯一标识符的 1 个 Token 的探测请求。通过对比供应商 API 的实际响应时间、缓存命中情况和计费回调，与 Layer 2 中的声明进行交叉验证。该探测成本极低，能为系统提供真实的、持续更新的实测数据。
-   **探测数据用于**：
    1.  计算“声明偏移量”——一个纯数学偏差百分比。
    2.  作为 Layer 5 软权重的附加输入，影响动态排序但不覆盖静态声明。
    3.  通过 Gossip 网络在社区节点间加密共享。

#### 🧮 Layer 3: 标准化计价与评估层

-   **痛点**：用户不想做算术题。
-   **确定性工程方案**：内部运行一个“预估成本推断引擎”。在请求发出前，它根据当前 Payload、历史缓存命中率，以及 Layer 2 的原始价格，动态计算出一个标量 `Estimated_Cost_USD`。它将复杂的计费模型瞬间坍缩为一个直观的美元数字。

#### 👤 Layer 4: 用户偏好与硬边界

-   **痛点**：用户有预算限制、数据合规红线。
-   **确定性工程方案**：基于 Hard Filter（硬性过滤规则）。用户在本地配置文件中定义约束，该层根据这些规则无情地砍掉所有不满足条件的供应商。被过滤的供应商连进入下一层的资格都没有。

#### 🧠 Layer 5: Agent 意图与动态偏好

-   **痛点**：不同任务需要最匹配的模型。
-   **确定性工程方案**：基于 Soft Weighting（柔性权重打分）。Agent 在 API 请求头中注入语义上下文。Sidecar 对剩余的候选模型执行确定性的多维向量点积打分。声明偏移量在此层作为权重修正因子参与计算——非覆盖、非惩罚、纯数学。

### 5.4 分布式韧性：熵增路由与抖动控制

为了避免全球所有 Sidecar 同时将流量涌入评分最高的单一供应商，FlowModus 在 Layer 5 引入了概率分布抽样机制。

-   **熵增路由 (Entropy-Weighted Routing)**：当 Layer 5 完成评分后，Sidecar 不是简单地返回第一名，而是将 Top-N 候选供应商的得分归一化为一个概率分布。系统以概率 P_i 选择供应商 i。这确保了即使两台不同的 Sidecar 运行完全相同的配置，它们最终发出的路由请求也是非确定性地分散的。
-   **平滑度系数 (Smoothness Coefficient)**：由 P99 延迟的标准差倒数定义，作为 Layer 2.5 探测日志中的客观记录项。用于辅助观察供应商 API 在高压下的稳定性，当前版本不进入 Layer 5 权重计算。

### 5.5 实时修正机制：双轨验证

FlowModus 建立了一套双轨验证体系，确保 Layer 2.5 始终掌握最接近物理现实的数据：

-   **定时探测 (Canary Probe)**：每小时一次的心跳测试，用于维护供应商的基础健康度和基础计费准确性。
-   **实时遥测 (Real-time Telemetry)**：当 Sidecar 处理真实的、高并发的业务请求时，它会记录实际的端到端延迟、Token 生成速度及并发下的缓存命中率。
-   **数据作用方式**：实测数据作为 Layer 2.5 的输入，生成声明偏移量。声明偏移量参与 Layer 5 的软权重计算。**实测数据永远不覆盖供应商的静态声明。静态声明保持可审查的原始状态，偏移量另行记录。**

### 5.6 算力审计：声明偏移量与用户自主判断

FlowModus 不代替用户判断供应商是否“诚信”。它只提供确定性的数据基础，把判断权交还用户：

-   **本地记录**：Sidecar 记录每一次请求中，供应商 API 返回的 `usage` 对象（`prompt_tokens`、`completion_tokens` 等）与供应商 JSON 声明的计费参数之间的对应关系。
-   **声明偏移量计算**：持续监测供应商 API 返回的用量数据是否存在异常波动。声明偏移量是一个纯数学量——声明值与实测值之间的偏差百分比。这个值自动计算、自动公开、不做任何评级。
-   **用户配置拦截**：用户可以在本地配置中设定自己的偏移量容忍阈值。当某供应商的声明偏移量超过用户设定的阈值时，Sidecar 可按用户预设规则采取降权或告警动作——**这是用户的决策，不是协议的判断。**
-   **审计维度**：所有审计基于时间戳窗口 + 模型名 + 请求用量特征的组合键进行匹配，不依赖任何需要供应商配合的自定义 HTTP 头部。

### 5.7 FlowModus 一致性测试

一致性测试是一套完全基于 Layer 2.5 探测数据的自动化客观数据呈现框架。该测试不由任何人工审核，而是由 Sidecar 持续进行的 1-Token 基准探测结果自动生成。

**测试指标**（均为客观数据，不做通过/不通过判断）：

| 测试项 | 呈现内容 | 数据来源 |
|:---|:---|:---|
| **计费精度** | 实际计费与声明计费的声明偏移量 | Layer 2.5 探测请求的账单回调 |
| **KV Cache TTL 漂移** | 实际 TTL 与声明的偏差值 | Layer 2.5 探测请求的缓存命中延迟 |
| **声明响应时间一致性** | P99 延迟与声明的偏差百分比 | Layer 2.5 探测请求的端到端延迟 |
| **语义路由标签准确度** | 探测任务结果与声明标签的匹配数据 | Layer 2.5 探测请求 + 基准任务评估 |

**关键设计原则**：

-   **只呈现数据，不做评判**：FlowModus 不设定任何“通过标准”。它只展示每个测试项的具体量化数据。
-   **完全自动化**：所有测试数据来自 Layer 2.5 的持续探测，无需任何人工干预。
-   **Gossip 网络共识**：测试结果通过 Gossip 协议在社区节点间共享。
-   **用户可配置**：用户可以在本地 Sidecar 配置中，自行设定自己的关注阈值。这是用户自己的判断，不是协议的结论。


## 第六章：与 Helix 生态的协同

作为 Helix 生态的调度规则定义者，FlowModus 的核心价值在与生态内其他项目协同工作时尤为凸显。本章描述应用层如何消费 FlowModus 提供的标准化数据——这些协同案例不构成 FlowModus 协议的必要组件，FlowModus 是完全独立的协议。

### 6.1 协同基础：标准化的供应商数据

Callosum 是一个独立的上下文内存管理器。在 FlowModus 出现之前，Callosum 的 KV Cache 优化参数需要硬编码配置。

FlowModus 的供应商 JSON 注册表，为 Callosum 提供了一个标准化的、可验证的、自动同步的数据源。Callosum 不再需要手动维护每个供应商的缓存参数，只需订阅 FlowModus 的 IPNS 链接，即可自动获取最新的供应商 KV Cache 能力声明。

这种协同遵循“编排优先，拒造实体”的哲学——FlowModus 只负责提供确定性的数据标准，不干预 Callosum 的决策逻辑。

### 6.2 确定性基准测试的协同价值

FlowModus 的 1-Token 基准探测为 Callosum 提供了持续验证供应商 KV Cache 声明的手段。通过定期发送探测请求，系统能够：

1.  验证缓存有效性：确认供应商的 KV Cache 在特定 TTL 内确实可用。
2.  检测缓存失效模式：发现在特定并发或时间段内缓存失效的情况。
3.  量化缓存收益：通过对比缓存命中与未命中的实际延迟和成本，量化 KV Cache 节约的 Token 量和美元成本。

这些数据以匿名形式在 FlowModus 的社区健康网络中共享——这是 FlowModus“不争市场，只定规则”哲学在工程层面的体现。

### 6.3 代码级协同示例

以下伪代码展示了 FlowModus Sidecar 与 Callosum 在 Helix 生态中的典型协同流程：

```python
# Anaphase 发起 LLM 请求时的协同决策流程

def optimize_llm_request(prompt: str, agent_role: str, cognitive_mode: str):
    # 1. Callosum: 根据认知模式确定缓存策略
    cache_strategy = callosum.get_cache_strategy(cognitive_mode)

    # 2. FlowModus: 根据供应商声明和缓存策略选择最优路由
    routing_decision = flowmodus_sidecar.route_request(
        prompt=prompt,
        agent_role=agent_role,
        cache_strategy=cache_strategy,
        user_constraints={"max_cost": 0.05, "region": "EU"}
    )

    # 3. Callosum: 将确定的供应商和缓存参数注入请求
    optimized_request = callosum.inject_cache_parameters(
        original_request=prompt,
        supplier=routing_decision.supplier,
        kv_cache_params=routing_decision.kv_cache_params
    )

    return optimized_request
```

### 6.4 生态定位：协议的纯粹性与应用的多样性

FlowModus 的白皮书和核心协议中，不依赖任何 Helix 生态的具体项目。FlowModus 是一个完全独立的、厂商中立的 LLM API 调度标准，可被任何 Agent 框架、任何 LLM 应用单独使用。

Helix 生态（如 Anaphase、Callosum、Cellrix）是 FlowModus 的应用层消费者——它们利用 FlowModus 提供的标准化数据，实现各自的优化目标。这种分层关系是“编排优先”哲学的直接体现：**协议定义标准，应用消费标准。**

### 6.5 基于实测数据的认知反馈

应用层消费者在处理真实业务流量时，持续监测每一个 Token 的往返。实测数据由 Sidecar 本地处理，生成声明偏移量，参与 Layer 5 权重修正。这一反馈回路不依赖任何单一应用的存在——任何接入 FlowModus 的 Agent 框架均可从中获益。

厂商的广告可以被市场部门操控，但一个正在本地运行、为自己的生存而优化成本的 Sidecar 实例只相信自己的检测数据。这种“基于物理现实的数据反馈”，是 FlowModus 数据真实性的基础。

### 6.6 技术极限的量化：长上下文衰减

应用层消费者的实测数据不仅能量化计费偏差，还能揭示那些难以从厂商公开文档中获取的技术边界。

以“长上下文中间信息检索衰减”为例——这是 Transformer 架构在超长上下文中存在的物理局限。部分厂商宣称支持超大上下文窗口时，其中间信息检索的召回率可能在实测中出现显著下降。

FlowModus 不对这种现象做任何“是欺诈还是正常物理限制”的判断。它只需将不同上下文深度下的实测召回率数据，作为供应商 JSON 中可选的扩展字段承载并公开。Agent 在任务规划时，可依据这些公开数据自行判断是否选择该模型处理长文档任务。

### 6.7 匿名数据共享与行业数据聚合

通过社区节点的自愿性数据共享，FlowModus 能够聚合生成行业级的算力数据视图。这些数据不包含任何用户的原始 Prompt，只基于本地聚合的统计数据进行发布：

-   **实时网络热图**：基于全球匿名节点反馈，实时显示不同地区、不同供应商的可用性和延迟分布。
-   **声明偏移量排名**：完全基于实测数据（而非厂商报价）的供应商声明与实际表现的偏差对比。
-   **缓存效能指数**：量化展示在何种业务场景下，启用 KV Cache 能带来最大的成本节约。

**数据共享的非对称设计**：

-   **不上报数据（data_sharing: off，默认）**：用户的 Sidecar 仍可被动接收 Gossip 网络中的聚合数据——只读，不写。隐私保护不意味着信息落后。
-   **主动上报数据（data_sharing: on，用户主动开启）**：Sidecar 仅上报三条聚合数据——可用性（0/1）、P50/P99 延迟、声明偏移量。结构层物理隔离任何敏感信息。数据共享的开关完全在用户手中。

**关于 Sybil 攻击的坦率声明**：v1.7 的去中心化数据聚合依赖于自愿上报，使用截尾均值和异常值过滤等统计方法增强抗污染能力。完全拜占庭容错的分布式数据聚合是后续阶段的研究课题。


## 第七章：协议维护者信托与渐进式去中心化

### 7.1 核心原则

FlowModus 遵循**循序渐进去中心化**路线：早期集权以快速迭代，成熟后分权以达成永生。治理层多签与执行层全网节点构成双重去中心化——协议的生存永远不绑定于任何单一实体或个人。

### 7.2 阶梯式多签治理

| 阶段 | 时间 | 签名结构 | 说明 |
|:---|:---|:---|:---|
| **阶段1：奠基人治理** | 0–6个月 | 单人签名 | 奠基人快速迭代协议。期间设有**心跳死手开关**：若连续90天无签名更新，全网 Sidecar 自动触发“奠基人失联警报”，保持现有路由规则继续运行的同时，开放社区平滑 Fork 权限。 |
| **阶段2：双人多签** | 6–12个月 | 2-of-3 | 引入 2 位独立委员与奠基人共同管理，跨出分权第一步。 |
| **阶段3：全球多签** | 12个月+ | M-of-N | 建立跨越多个大洲、互不隶属的极客与安全研究者委员会（如 5-of-7）。 |

协议更新频率极低，仅需异步签名，时差和跨国协作成本不会影响系统的执行效率。

### 7.3 委员身份与公信

-   **匿名招募**：委员使用全新海外匿名账号，需具备半年以上独立可查的技术动态。发言风格、活跃时区、IP 地域完全独立，委员间不互相关注，从物理隔离层面抵御巨头女巫攻击。
-   **提名制与社区否决**：委员由奠基人或现任委员会提名，社区有权提出合理质疑与否决。第一代委员采用纯荣誉制。
-   **公开信息**：仅公示委员的 Ed25519 公钥与匿名账号 ID。绝不公开真人姓名、国家、联系方式等现实世界关联隐私。

### 7.4 多签验证的极简工程实现

FlowModus 拒绝使用复杂且容易引入漏洞的密码学聚合方案。多签机制采用最稳健的**签名数组与本地阈值判定**：

1.  每位委员在离线设备生成专属 Ed25519 私钥，绝不触网。
2.  JSON 注册表中包含一个 `signatures` 数组。
3.  Sidecar 客户端硬编码 N 个委员的公钥列表。在拉取到更新后，Sidecar 在本地遍历验证。只要有效的独立签名数量达到 M 的阈值，更新即刻生效。

此方案无需任何高级密码学库，一个 `for` 循环即可实现，工程上绝对安全。

### 7.5 四层不死兜底

| 层级 | 机制 | 效果 |
|:---|:---|:---|
| **治理层** | 委员跨国家且物理隔离。若多数委员被恶意控制或胁迫，全球社区可直接 Fork 注册表，在本地 Sidecar 中替换信任公钥列表。 | 治理不可消灭 |
| **代码层** | IPFS 永久 CID + IPNS 动态寻址。任何单一中心化代码托管平台的封禁，无法阻断协议的运行与分发。 | 代码不可删除 |
| **执行层** | 全球 Sidecar 节点均为本地独立运行，完全掌握控制权。节点有权主动拒绝未经验签或被社区标记为恶意的协议更新。 | 节点不可强制 |
| **法律层** | 协议不控制流量、不触碰资金。后期域名和公共基础设施由离岸非营利主体或去中心化自治组织代持。 | 个人不可追责 |
