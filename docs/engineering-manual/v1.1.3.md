## §0. 工程铁律 (Engineering Iron Laws)

**违反本章任一条的代码，不得合入主分支。本章无例外。**

### 0.0 最小作用量原理 (Principle of Least Action / Zero Waste)

系统绝不主动产生任何试探性动作。

- 废除一切定时轮询和主动探测。
- Sidecar 处于绝对的事件驱动和惰性求值状态。无请求时 CPU 占用为零。
- 所有对外部世界状态的感知，**100% 寄生于真实业务数据流**中。
- **唯一例外**：用户显式触发的按需探测，且此操作的成本必须在 UI 上提前告知用户。
- **冷启动补充**：首次请求即首次感知。所有供应商初始状态为 `UNKNOWN`。首个真实请求基于 Layer 2 纸面声明选出理论最优供应商。若失败，Sidecar 执行无感重试，不向 Agent 暴露错误。

### 0.1 确定性优先 (Determinism First)

任何给定相同输入的函数或模块，必须产生比特级相同的输出。禁止引入以下非确定性来源：

- 禁止：`random`（除非显式传入种子）
- 禁止：`time.now()` 作为计算输入（仅允许用于日志时间戳）
- 禁止：依赖迭代顺序的 `dict`/`set` 遍历（必须排序后遍历）
- 禁止：未固定版本的依赖（`pyproject.toml` 必须锁定到 patch 版本，`uv.lock` 必须提交）

**允许的非确定性**：网络 I/O 的超时、操作系统信号处理。这些必须在边界层隔离，不得渗透进核心路由逻辑。

**确定性补充条款**：确定性指**单节点可复现性**——同一节点在相同输入下必须产生相同输出。全局去相关通过 **Sidecar 实例 ID** 实现：不同节点的相同请求产生不同路由选择，避免全球同步效应。此机制在 Layer 5 中工程化实现。

### 0.2 纯函数优先 (Pure Function First)

所有路由决策逻辑必须以纯函数实现。纯函数定义：

- 输出仅取决于输入参数
- 不读取全局状态（全局常量除外）
- 不修改输入参数
- 不产生副作用（不写文件、不发网络请求、不打印日志）

副作用集中在 Sidecar 生命周期的边界层（启动初始化、优雅关闭、被动遥测采集）。

### 0.3 无锁优先 (Lock-Free First)

Sidecar 内部不使用线程锁、互斥量、信号量。并发模型采用消息传递：

- 控制面与数据面是独立的事件循环或进程
- 数据面内部，每个管线层是独立的状态机
- 层间通过不可变消息对象通信
- 使用 `asyncio` 协程 + 队列，不使用 `threading`

**理由**：死锁是分布式系统最隐蔽的杀手。消息传递消除死锁可能性。

### 0.4 禁止阻塞 (No Blocking)

数据面的主事件循环中任何操作不得阻塞超过 1ms。网络 I/O 必须异步。文件 I/O 必须在控制面或启动阶段完成，运行时只允许追加式日志写入。

### 0.5 Schema 强制 (Schema Enforcement)

所有跨模块数据结构必须有 Protobuf 定义（`schemas/` 目录）。模块间通信只允许使用生成的类型桩。禁止：

- 裸 `dict` 作为模块间接口
- 裸 `str` 传递 JSON（必须在防腐层完成序列化/反序列化）
- 裸 `bytes` 传递未定义结构的数据

### 0.6 边界校验 (Boundary Validation)

所有来自 Sidecar 外部的数据（供应商 JSON、环境变量、HTTP 请求头）必须在进入核心逻辑前完成校验。校验规则由 JSON Schema 或 Protobuf 定义。校验失败的数据不得进入任何管线层。

### 0.7 依赖最小化 (Minimal Dependencies)

依赖引入原则：

- 密码学：仅 `cryptography` 库（Ed25519）
- 序列化：仅 `protobuf`
- HTTP 客户端：仅 `httpx`（异步）
- HTTP 服务器：仅 `aiohttp.web`（轻量级，不含框架包袱）
- 禁止：引入 ORM、引入重型 Web 框架 (FastAPI/Flask/Django)、引入任何需要守护进程的依赖

每个新依赖在引入前必须在 PR 中说明不可替代性。

### 0.8 命名即文档 (Naming as Documentation)

遵循 Google 命名规范（详见附录 A）。命名必须自解释。禁止缩写（已在本手册约定表中列出的除外）。函数超过 20 行必须拆分。**禁止模块名与所属包重名。**

## §2. Sidecar 生命周期 (Sidecar Lifecycle)

### 2.1 状态机定义

```
                    ┌─────────────┐
                    │   INIT      │
                    └──────┬──────┘
                           │ start()
                           ▼
                    ┌─────────────┐
                    │  BOOTSTRAP  │
                    └──────┬──────┘
                           │ 成功
                           ▼
                    ┌─────────────┐
                    │  READY      │◄──────── 正常运行状态
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │DEGRADED │ │ FREEZING │ │SHUTTING  │
       └────┬─────┘ └──────────┘ └────┬─────┘
            │                          │
            ▼                          ▼
       ┌──────────┐              ┌──────────┐
       │  READY   │              │  DEAD    │
       └──────────┘              └──────────┘
```

**状态说明**：

| 状态 | 含义 | 路由行为 |
|:---|:---|:---|
| `INIT` | 进程启动，未加载任何配置 | 无 |
| `BOOTSTRAP` | 控制面正在拉取和验证注册表 | 无 |
| `READY` | 正常服务 | 完整五层管线 |
| `DEGRADED` | 部分供应商不可用 | 使用缓存注册表 + 熔断不可用供应商 |
| `FREEZING` | 奠基人失联警报触发 | 使用冻结快照，拒绝新供应商 |
| `SHUTTING_DOWN` | 收到 SIGTERM/SIGINT | 处理进行中请求，拒绝新请求 |
| `DEAD` | 进程退出 | 无 |

### 2.2 状态转换规则

```python
# src/flowmodus/lifecycle.py

from enum import Enum, auto
from dataclasses import dataclass
from typing import Optional

class SidecarState(Enum):
    INIT = auto()
    BOOTSTRAP = auto()
    READY = auto()
    DEGRADED = auto()
    FREEZING = auto()
    SHUTTING_DOWN = auto()
    DEAD = auto()
```

**转换规则（确定性）**：

| 从 | 到 | 触发条件 |
|:---|:---|:---|
| `INIT` | `BOOTSTRAP` | 调用 `start()` |
| `BOOTSTRAP` | `READY` | 控制面全部成功 |
| `BOOTSTRAP` | `FREEZING` | 控制面拉取失败 + 有本地缓存 + 距上次成功 > 90 天 |
| `BOOTSTRAP` | `DEAD` | 控制面拉取失败 + 无本地缓存 |
| `READY` | `DEGRADED` | 真实请求连续失败超过用户配置阈值 |
| `DEGRADED` | `READY` | 无感重试或复健采样确认恢复 |
| `READY`/`DEGRADED` | `FREEZING` | 收到恶意更新通知或手动触发 |
| `FREEZING` | `READY` | 手动解除 + 重新验签通过 |
| `READY`/`DEGRADED`/`FREEZING` | `SHUTTING_DOWN` | SIGTERM/SIGINT |
| `SHUTTING_DOWN` | `DEAD` | 进行中请求完成或超时 |

### 2.3 控制面与数据面分离

Sidecar 内部严格划分为两个独立的事件循环：

**控制面 (Control Plane)**：
- 负责：IPNS 拉取、JCS 规范化、Ed25519 验签、JSON → Protobuf 转换
- 运行频率：启动时 + 每小时刷新
- 它慢，但不与用户请求竞争资源

**数据面 (Data Plane)**：
- 负责：HTTP 代理、五层路由管线、被动遥测采集
- 只读内存中的 Protobuf 快照，O(1) 速度
- 零阻塞，零 I/O

**通信机制**：控制面验证成功后，构建新的只读快照，通过原子引用替换交给数据面。

### 2.4 启动流程

```
1. 加载本地配置 (环境变量 + .env + bias.yaml)
2. 验证本地配置 Schema
3. 【控制面】拉取 IPNS 注册表
4. 【控制面】Ed25519 验签（在原始 bytes 上）
5. 【控制面】防腐层：JSON → Protobuf Message
6. 加载缓存的被动遥测数据到内存只读快照
7. 初始化数据面（五层管线 + HTTP 代理）
8. 初始化 Gossip 监听（若开启数据共享）
9. 就绪 → 开始接受代理请求
```

**防腐层实现**：使用 `google.protobuf.json_format.ParseDict()` 完成 JSON dict → Protobuf Message 的类型安全转换。不手写字段映射代码。

### 2.5 优雅关闭

```python
# src/flowmodus/lifecycle.py

async def shutdown(sidecar, signal_received: signal.Signals) -> None:
    sidecar.state = SidecarState.SHUTTING_DOWN
    await asyncio.wait_for(
        sidecar.drain_inflight_requests(),
        timeout=sidecar.config.shutdown_timeout_sec
    )
    sidecar.telemetry_store.flush()
    sidecar.state = SidecarState.DEAD
```

**流式中断补充**：进行中的 SSE 流式请求被中断时，Sidecar 在关闭前发送最后一个 `assistant` 消息片段，附带 `finish_reason: "flowmodus_interrupted"` 标记，供 Agent 判断是否需要续传。此特性在 v1.7 中预留接口，完整实现在 v2.0。

### 2.6 周期任务

| 任务 | 周期 | 说明 |
|:---|:---|:---|
| 注册表刷新 | 每小时 | 控制面负责 |
| 被动遥测聚合 | 每次真实请求后 | 数据面负责，零额外网络成本 |
| Gossip 状态跃迁广播 | 仅当供应商状态跃迁时 | 平时静默 |
| 心跳日志 | 每分钟 | 仅记录自身状态 |

**禁止**：任何定时主动探测。禁止向供应商发送非用户请求的 HTTP 请求。


## §3. 五层路由管线 (The 5-Layer Deterministic Routing Pipeline)

### 3.1 架构总览

```
用户请求
   │
   ▼
┌────────────────────────────────────────────────┐
│ Layer 1: 协议与度量规范                         │
│ 纯函数: normalize_request() + calculate_ste()    │
└────────────────────────────────────────────────┘
   │
   ▼
┌────────────────────────────────────────────────┐
│ Layer 2: 供应商原始模型                         │
│ 数据源: 控制面验证后的 Protobuf 快照             │
└────────────────────────────────────────────────┘
   │
   ▼
┌────────────────────────────────────────────────┐
│ Layer 3: 标准化计价与评估                       │
│ 纯函数: estimate_cost()                         │
└────────────────────────────────────────────────┘
   │
   ▼
┌────────────────────────────────────────────────┐
│ Layer 4: 用户偏好与硬边界                       │
│ 纯函数: apply_hard_filters()                    │
│ 输入: CostEstimate[] + UserConstraints          │
│       + bias.yaml (用户主观偏好)                │
│       + DeviationSnapshot (内存快照)            │
│       + priority_groups (渠道优先级)            │
└────────────────────────────────────────────────┘
   │
   ▼
┌────────────────────────────────────────────────┐
│ Layer 5: Agent 意图与动态偏好                   │
│ 纯函数: score_and_entropy_sample()              │
│ 输入: EligibleSupplier[] + AgentContext         │
│       + instance_id (局部变异因子)              │
│       + bias_score (用户主观加分)               │
│       + 复健采样概率                            │
└────────────────────────────────────────────────┘
   │
   ▼
路由决策 → 代理层发出请求
```

### 3.6 无感重试与渠道保护协议

#### 3.6.1 供应商健康状态分类

| 状态 | 含义 | 进入条件 | 离开条件 |
|:---|:---|:---|:---|
| `HEALTHY` | 正常服务 | 2xx | 触发 `RETRYABLE_ERROR` |
| `DEGRADED` | 暂时不可用 | 429, 5xx, 连接超时 | 复健采样成功 |
| `TERMINAL` | 不可重试 | 401, 403, 400 | 用户手动修复 |

#### 3.6.2 错误状态映射表

```python
def classify_http_error(status_code: int, exception: Exception = None) -> str:
    if 200 <= status_code < 300:
        return "success"
    
    if status_code in (429, 500, 502, 503, 504):
        return "retryable"
    
    if isinstance(exception, (ConnectTimeout, ConnectError)):
        return "retryable"
    
    if status_code in (401, 403, 400):
        return "terminal"
    
    return "retryable"
```

#### 3.6.3 无感重试算法

1. 仅当 `retryable` 时触发
2. 标记该端点为 `DEGRADED`
3. 最大重试 3 次
4. 每次重试重新经过 Layer 4 和 Layer 5
5. 重试耗尽 → 向 Agent 返回错误
6. Agent 不感知底层切换

#### 3.6.4 概率性复健采样

REHABILITATION_PROBABILITY = 0.001  # 0.1%

## §5. 被动遥测与按需探测 (Passive Telemetry & On-Demand Probe)

### 5.1 核心原则

- **零主动探测**：Sidecar 不发送任何非用户请求的 HTTP 请求
- **被动寄生**：所有供应商性能数据从真实业务请求中提取
- **按需探测**：仅用户显式触发，且提前告知成本

**零负担 TTFT 采集**：采用 Wrapper Pattern 包装供应商响应流，在第一个 chunk 到达时记录时间戳。不产生任何额外 CPU 轮询。

```python
class TTFTTimedStream:
    def __init__(self, response_stream, start_time: float):
        self._stream = response_stream
        self._start_time = start_time
        self._first_chunk = True
        self.ttft_ms: Optional[float] = None
    
    async def __aiter__(self):
        async for chunk in self._stream:
            if self._first_chunk:
                self.ttft_ms = (time.monotonic() - self._start_time) * 1000
                self._first_chunk = False
            yield chunk
```

**结算偏移量滑动窗口审计**：

```python
SETTLEMENT_WINDOW_SIZE = 100
SETTLEMENT_ALERT_THRESHOLD = 10.0  # 百分比

def calculate_settlement_deviation(window: list[float]) -> float:
    """滑动窗口加权平均。越新样本权重越高。纯函数。"""
    if not window:
        return 0.0
    
    total_weight = 0.0
    weighted_sum = 0.0
    for i, deviation in enumerate(window):
        weight = i + 1
        weighted_sum += deviation * weight
        total_weight += weight
    
    return weighted_sum / total_weight
```

报警基于窗口均值而非单次值。防止因换行符/Emoji 等分词差异导致的误报。

## §6. 数据共享协议 (Data Sharing Protocol)

### 6.1 核心原则

- 默认关闭 (`data_sharing: off`)
- 仅上报三条聚合数据：可用性、延迟、声明偏移量
- 结构层物理隔离敏感信息

### 6.2 状态跃迁广播模型

Gossip 网络绝不发送定时心跳。仅在供应商状态发生关键跃迁时广播：

| 跃迁 | 广播内容 | 触发条件 |
|:---|:---|:---|
| 遭遇疼痛 (Fault) | `[DOWN: X]` | `HEALTHY` → `DEGRADED` |
| 伤愈复苏 (Recovery) | `[UP: X]` | `DEGRADED` → `HEALTHY` |
| 发现宝藏 (Golden Cache) | `[HINT: X Cache]` | 罕见高价值缓存命中 |

平时绝对静默。只在物理状态改变时同步 Delta。

### 6.3 本地免疫系统

Gossip 数据标记为**弱提示**。

- 接收到的 `[DOWN]` → 暂时降权该供应商
- 若后续真实请求成功 → Gossip 来源节点进入本地黑名单
- 该节点此后所有消息被忽略

每个 Sidecar 独立判断，不依赖全局共识。

### 6.4 节点身份

- 每次启动生成临时 Ed25519 密钥对
- 临时公钥的 SHA-256 哈希作为 `instance_id`
- 关闭时销毁临时私钥

### 6.5 接收模式

| 模式 | data_sharing | 上报 | 接收聚合数据 |
|:---|:---|:---|:---|
| 纯消费者 | `off` (默认) | 否 | 是 |
| 社区贡献者 | `on` | 是 | 是 |

## 附录 B：反模式清单 (Anti-Pattern Catalog)

**违反以下任一条的代码，不得合入主分支。**

### B.1 架构反模式

| 反模式 | 正确做法 |
|:---|:---|
| 裸 dict 作为模块接口 | Protobuf 类型或 dataclass |
| 裸 JSON string 传递 | 结构化对象，序列化在边界层 |
| 深层嵌套回调 | async/await 扁平化 |
| 跨层直接访问数据库 | 内存快照注入 |

### B.2 并发反模式

| 反模式 | 正确做法 |
|:---|:---|
| 使用 `threading` | `asyncio` |
| 使用 `time.sleep()` | `await asyncio.sleep()` |
| 共享可变状态 | 消息队列 + 不可变对象 |
| 无超时的网络请求 | 显式 timeout |

### B.3 密码学反模式

| 反模式 | 正确做法 |
|:---|:---|
| 自行实现加密原语 | `cryptography` 库 |
| 私钥触网 | 离线生成，U 盘传递 |
| 非标准签名方案 | 标准 Ed25519 |
| 硬编码私钥 | 代码中只有公钥 |

### B.4 依赖反模式

| 反模式 | 正确做法 |
|:---|:---|
| 引入重型 Web 框架 | `aiohttp.web` 或 `asyncio.start_server` |
| 引入 ORM | 原生 SQL |
| 未锁定版本 | `uv.lock` 提交 |

### B.5 命名反模式

| 反模式 | 正确做法 |
|:---|:---|
| `def proc(x)` | `def process_request(request: RawRequest)` |
| `data["key"]` | `supplier.supplier_id` |
| `ctx`, `cfg`, `val` | `context`, `config`, `value` |

### B.6 哲学反模式

| 反模式 | 说明 |
|:---|:---|
| 定时轮询 / 主动探测 | 违反铁律 0.0 |
| 协议做判断 | 只呈现数据，不评级 |
| 协议做推荐 | 不推荐供应商 |
| 隐藏复杂性 | 数据流显式化 |
| 过度抽象 | 继承链 ≤ 3 层 |

### B.7 屏蔽"幻觉代码" (Anti-Hallucination Code)

**禁止**在 Sidecar 内部对供应商响应进行任何"智能修复"、"猜测补全"或"格式纠正"。

**理由**：FlowModus 的使命是确定性。如果供应商返回了损坏的 JSON 或不完整的流，Sidecar 必须直接报错并触发重试（若为可重试错误），绝对禁止尝试"帮供应商修好 JSON"。修数据是供应商的责任，不是 Sidecar 的。
