# FlowModus

**LLM API 调度的确定性协议层**
*去中心化 · 厂商中立 · 社区共建*

---

### ⚡ 概述

FlowModus 是位于你的 Agent 与 LLM 供应商之间的**密码学验证、确定性路由层**。它用本地优先的 Sidecar 替代不透明、抽租式的传统网关，执行一套数学上严密的五层管线。

*   **问题**：传统网关基于供应商利润路由，而非用户意图。计费不透明、延迟不确定、算力浪费。
*   **方案**：一个零浪费的 Sidecar，强制执行用户定义的约束，通过 Ed25519 验证每一份路由表，所有遥测数据来自真实流量——绝不主动探测。
*   **结果**：可验证的逐 Token 成本归因、完整的数据主权、遵循*你的*规则而非供应商规则的路由。

## 🧬 五层确定性路由管线

每个请求经过五个不可变的层级。没有一步是可选的，没有一个决策是随意的。

1.  **规范化 (STE)**——将供应商特定的 Token 转换为标准 Token 当量。
2.  **原始注册表**——Ed25519 签名、IPFS/IPNS 分发的供应商声明。
3.  **成本推断**——结合缓存提示和声明偏移量的实时价格估算。
4.  **硬性过滤**——在本地强制执行预算、区域、数据驻留和用户自定义偏好。
5.  **熵增路由**——通过实例 ID 注入确定性抖动，防止全球群体效应。

## 🔀 三种调用模式

FlowModus 支持三种路由模式，通过请求体中的 `model` 参数选择。

| 模式 | `model` 值 | 行为 |
|:---|:---|:---|
| **手动模式** | `"deepseek-chat"` | 直接路由到指定模型，无管线开销 |
| **组级路由** | `"group:fast-lane"` | 在用户定义的分组内按优先级和权重路由 |
| **自动模式** | `"auto"` | 完整五层管线：成本估算、硬性过滤、熵增采样 |

## 🪨 工程铁律（来自工程指导手册）

*这些是不可协商的。违反任一条的代码不会被合入。*

0.  **最小作用量原理**——零主动轮询。所有遥测均为寄生式。零浪费。
1.  **确定性优先**——相同输入 → 比特级相同输出，通过实例 ID 进行全局去相关。
2.  **纯函数优先**——所有路由逻辑无副作用。
3.  **无锁优先**——消息传递优于共享内存。无死锁。
4.  **禁止阻塞**——仅异步 I/O。控制面与数据面分离。
5.  **Schema 强制**——模块间使用 Protobuf。禁止裸 `dict` 或 JSON。
6.  **依赖最小化**——无 Web 框架、无 ORM、无守护进程。
7.  **命名即文档**——Google 风格。命名自解释。

## 🏗 架构

FlowModus 以**本地 Sidecar** 方式运行（`localhost:8080`）。在完整的 Helix 部署中，调用链为：

```
Anaphase‑Helix → Callosum → Tuck → FlowModus → LLM API
```

供应商注册表从 IPFS 拉取，使用协议根密钥验证。API 密钥在边缘注入——永不出内存，永不出现在日志中，永不通过 Gossip 网络传输。

## ⚠️ 当前临时方案（Cellrix 集成前）

**重要提示**：在 Cellrix 集成提供引导式 UI 之前，以下临时配置方法正在使用。**请务必记得**在后续用计划中的 `config.yaml` 和引导式设置替换掉它们。

### 🔧 环境变量（临时）
目前，必须手动导出环境变量才能启动 Sidecar：
```bash
export FLOWMODUS_LOCAL_REGISTRY=~/.flowmodus/local_registry.json
export FLOWMODUS_API_KEY_DEEPSEEK=sk-...
export FLOWMODUS_API_KEY_TUCK_LOCAL=sk-...
export FLOWMODUS_PROXY_PORT=8080
uv run flowmodus
```
*为何是临时的*：它强迫用户记忆变量名并输入冗长的命令。  
*计划替代方案*：一个单独的 `config.yaml` 文件（见下文），最终将使用 Cellrix 引导表单。

### 🔀 模型名称替换（临时）
在自动模式中，管线目前会**在 `proxy.py` 内部**将请求体中的 `model` 字段替换为实际的供应商模型 ID（例如 `Qwen2.5.1-Coder-7B-Instruct-Q4_K_M.gguf`）。  
*为何是临时的*：它修改了用户的原始请求，违反了不可变性原则。  
*计划替代方案*：将此逻辑移入 `lifecycle.py`，使 `proxy.py` 接收到完整的决策对象，而绝不修改原始请求体。

### 🗺️ Cellrix 更新后的后续步骤

1. **将配置迁移至 `~/.flowmodus/config.yaml`**——一个结构化、自解释的文件，取代分散的环境变量。
2. **修复模型名称替换**——将逻辑从 `proxy.py` 移至 `lifecycle.py`，以保持请求的不可变性。
3. **验证 Callosum → Tuck → FlowModus 链**——确保完整的 Helix 技术栈端到端运行。
4. **完成组级路由模式的实现**——启用 `model="group:..."` 路由。
5. **添加流式响应支持**——正确处理 SSE 流。
